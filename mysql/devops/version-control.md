# Version Control — Schema Versioning, Diff Tools, and DDL Review for MySQL

## Overview

Version-controlling a MySQL database schema is fundamentally different from version-controlling application code. Schemas are stateful: you cannot simply "check out" a previous version without understanding the data transformations required to move between states. Effective schema version control requires a strategy for tracking DDL changes, comparing schemas across environments, managing seed data, and reviewing changes before they reach production.

Two dominant approaches exist: numbered sequential migrations (each file represents a forward change) and diff-based approaches (compare desired state against current state and generate the delta). Most teams use numbered migrations for their explicitness and auditability, while diff-based tools are useful for validation and drift detection.

This skill covers schema versioning strategies, snapshot tooling, schema comparison, seed data management, and team workflows for reviewing database changes.

## Key Concepts

- **Schema snapshot**: A point-in-time DDL export of the database structure without data, typically generated by `mysqldump --no-data`.
- **Numbered migration**: A sequentially versioned SQL file (e.g., `V001_create_users.sql`) that moves the schema from state N to state N+1.
- **Diff-based migration**: Auto-generated DDL by comparing a desired schema (model) against the current database state.
- **Schema drift**: Untracked divergence between the expected schema (in version control) and the actual schema in a running database.
- **Seed data**: Reference or lookup data that is part of the schema definition (e.g., country codes, status enums stored in tables).
- **Baseline**: The initial schema snapshot from which all future migrations are tracked.

## Numbered Migration Strategy

### Directory structure

```
db/
  migrations/
    V001__baseline.sql
    V002__add_orders_table.sql
    V003__add_user_email_index.sql
    V004__add_status_column_to_orders.sql
    V005__create_audit_log_table.sql
  rollbacks/
    U002__drop_orders_table.sql
    U003__drop_user_email_index.sql
    U004__remove_status_column_from_orders.sql
    U005__drop_audit_log_table.sql
  seeds/
    R001__countries.sql
    R002__currencies.sql
  schema/
    current_schema.sql          # Latest full schema snapshot
```

### Migration file template

```sql
-- Migration: V004__add_status_column_to_orders.sql
-- Author: dba-team
-- Date: 2024-11-15
-- Description: Add status column to orders table for order tracking
-- Jira: DB-1234
-- Rollback: U004__remove_status_column_from_orders.sql

-- Pre-check: verify column does not already exist
SELECT COUNT(*) INTO @col_exists
FROM information_schema.COLUMNS
WHERE TABLE_SCHEMA = DATABASE()
  AND TABLE_NAME = 'orders'
  AND COLUMN_NAME = 'status';

SET @sql = IF(@col_exists = 0,
  'ALTER TABLE orders ADD COLUMN status ENUM("pending","processing","shipped","delivered") DEFAULT "pending" AFTER total',
  'SELECT "Column already exists, skipping" AS info');
PREPARE stmt FROM @sql;
EXECUTE stmt;
DEALLOCATE PREPARE stmt;
```

### Corresponding rollback

```sql
-- Rollback: U004__remove_status_column_from_orders.sql
ALTER TABLE orders DROP COLUMN status;
```

## Schema Snapshots with mysqldump

### Generating a schema-only snapshot

```bash
# Full schema snapshot (no data)
mysqldump -h db1 -u root -p \
  --no-data \
  --routines \
  --triggers \
  --events \
  --skip-comments \
  --skip-dump-date \
  --databases mydb \
  > schema/current_schema.sql

# Normalized for clean diffs (remove auto-increment counters, timestamps)
mysqldump -h db1 -u root -p \
  --no-data \
  --skip-comments \
  --skip-dump-date \
  --compact \
  mydb | \
  sed 's/ AUTO_INCREMENT=[0-9]*//' \
  > schema/current_schema.sql
```

### Automating snapshots on migration

```bash
#!/bin/bash
# post-migrate-snapshot.sh
# Run after every migration to keep the schema snapshot current

DB_HOST="${1:-localhost}"
DB_NAME="${2:-mydb}"
SCHEMA_FILE="schema/current_schema.sql"

mysqldump -h "$DB_HOST" -u root -p"$MYSQL_PASSWORD" \
  --no-data --routines --triggers --events \
  --skip-comments --skip-dump-date \
  "$DB_NAME" | \
  sed 's/ AUTO_INCREMENT=[0-9]*//' \
  > "$SCHEMA_FILE"

git add "$SCHEMA_FILE"
echo "Schema snapshot updated: $SCHEMA_FILE"
```

## Schema Comparison Tools

### mysqldiff (MySQL Utilities)

```bash
# Compare two live databases
mysqldiff --server1=root:pass@localhost \
          --server2=root:pass@staging:3306 \
          mydb:mydb \
          --difftype=sql \
          --force

# Compare a schema file against a live database
# (requires loading the file into a temp database first)
mysql -u root -p -e "CREATE DATABASE IF NOT EXISTS temp_compare"
mysql -u root -p temp_compare < schema/current_schema.sql
mysqldiff --server1=root:pass@localhost \
          --server2=root:pass@localhost \
          temp_compare:mydb \
          --difftype=sql
mysql -u root -p -e "DROP DATABASE temp_compare"
```

### pt-table-checksum (Percona)

```bash
# Verify schema and data consistency across replicas
pt-table-checksum \
  --host=primary \
  --user=ptcheck \
  --password=secret \
  --databases=mydb \
  --tables=users,orders \
  --check-replication-filters \
  --replicate=percona.checksums \
  --no-check-binlog-format

# Check results
pt-table-checksum \
  --host=primary \
  --replicate-check-only \
  --replicate=percona.checksums
```

### Manual schema diff with SQL

```sql
-- Find tables in dev but not in prod
SELECT TABLE_NAME FROM information_schema.TABLES
WHERE TABLE_SCHEMA = 'dev_db'
  AND TABLE_NAME NOT IN (
    SELECT TABLE_NAME FROM information_schema.TABLES
    WHERE TABLE_SCHEMA = 'prod_db'
  );

-- Find column differences between environments
SELECT
    d.TABLE_NAME,
    d.COLUMN_NAME,
    d.COLUMN_TYPE AS dev_type,
    p.COLUMN_TYPE AS prod_type
FROM information_schema.COLUMNS d
LEFT JOIN information_schema.COLUMNS p
  ON d.TABLE_NAME = p.TABLE_NAME
  AND d.COLUMN_NAME = p.COLUMN_NAME
  AND p.TABLE_SCHEMA = 'prod_db'
WHERE d.TABLE_SCHEMA = 'dev_db'
  AND (p.COLUMN_TYPE IS NULL OR d.COLUMN_TYPE != p.COLUMN_TYPE);

-- Find index differences
SELECT
    d.TABLE_NAME,
    d.INDEX_NAME,
    GROUP_CONCAT(d.COLUMN_NAME ORDER BY d.SEQ_IN_INDEX) AS dev_columns
FROM information_schema.STATISTICS d
WHERE d.TABLE_SCHEMA = 'dev_db'
  AND CONCAT(d.TABLE_NAME, '.', d.INDEX_NAME) NOT IN (
    SELECT CONCAT(TABLE_NAME, '.', INDEX_NAME)
    FROM information_schema.STATISTICS
    WHERE TABLE_SCHEMA = 'prod_db'
  )
GROUP BY d.TABLE_NAME, d.INDEX_NAME;
```

## Seed Data Management

### Idempotent seed files

```sql
-- seeds/R001__countries.sql
-- Repeatable: re-run safely when data changes
INSERT INTO countries (code, name) VALUES
  ('US', 'United States'),
  ('GB', 'United Kingdom'),
  ('DE', 'Germany'),
  ('JP', 'Japan')
ON DUPLICATE KEY UPDATE name = VALUES(name);
```

### Separate seed data from test fixtures

```
db/seeds/
  reference/         # Deployed to all environments
    countries.sql
    currencies.sql
    timezones.sql
  development/       # Dev-only sample data
    sample_users.sql
    sample_orders.sql
  test/              # CI test fixtures
    fixtures.sql
```

### Exporting seed data

```bash
# Export specific reference tables as INSERT statements
mysqldump -h db1 -u root -p \
  --no-create-info \
  --skip-triggers \
  --complete-insert \
  --skip-comments \
  --skip-dump-date \
  --where="1=1 ORDER BY id" \
  mydb countries currencies \
  > seeds/reference/lookup_data.sql
```

## Branching Strategies for Database Changes

### Trunk-based with migration sequencing

```
main
  V100__feature_a_table.sql     (merged first)
  V101__feature_b_column.sql    (merged second)
  V102__feature_a_index.sql     (merged third)
```

Rules:
- Migration version numbers are assigned at merge time, not at branch creation.
- Use a placeholder naming scheme in feature branches (e.g., `VNEXT__description.sql`).
- A CI check renumbers migrations to the next available version on merge.

### Conflict resolution script

```bash
#!/bin/bash
# renumber-migration.sh — Assign next version number at merge
MIGRATION_DIR="db/migrations"
LAST_VERSION=$(ls "$MIGRATION_DIR"/V*.sql 2>/dev/null | \
  sed 's/.*V\([0-9]*\)__.*/\1/' | sort -n | tail -1)
NEXT_VERSION=$(printf "%03d" $((LAST_VERSION + 1)))

for f in "$MIGRATION_DIR"/VNEXT__*.sql; do
  NEW_NAME=$(echo "$f" | sed "s/VNEXT/V${NEXT_VERSION}/")
  mv "$f" "$NEW_NAME"
  NEXT_VERSION=$(printf "%03d" $((10#$NEXT_VERSION + 1)))
done
```

## DDL Review Process

### Pull request checklist for schema changes

```markdown
## DDL Change Review Checklist
- [ ] Migration is idempotent (safe to re-run)
- [ ] Rollback file exists and has been tested
- [ ] Schema snapshot updated (`schema/current_schema.sql`)
- [ ] No implicit data type changes (e.g., VARCHAR length reduction)
- [ ] Large table? Specified ALGORITHM/LOCK or using pt-osc/gh-ost
- [ ] Foreign key impact assessed (CASCADE/RESTRICT)
- [ ] Index naming follows convention (`idx_tablename_columns`)
- [ ] Character set is utf8mb4 (not utf8/utf8mb3)
- [ ] Reviewed by DBA team
- [ ] Tested on staging with production-sized data
```

### Automated schema review in CI

```bash
#!/bin/bash
# ci-schema-review.sh
ERRORS=0

for f in db/migrations/V*.sql; do
  # Check for dangerous operations
  if grep -qi "DROP TABLE" "$f"; then
    echo "WARNING: $f contains DROP TABLE"
    ERRORS=$((ERRORS + 1))
  fi
  if grep -qi "TRUNCATE" "$f"; then
    echo "WARNING: $f contains TRUNCATE"
    ERRORS=$((ERRORS + 1))
  fi
  if grep -qi "utf8[^m]" "$f" | grep -v "utf8mb4"; then
    echo "ERROR: $f uses utf8 instead of utf8mb4"
    ERRORS=$((ERRORS + 1))
  fi
  # Ensure ALGORITHM is specified for ALTER TABLE
  if grep -qi "ALTER TABLE" "$f" && ! grep -qi "ALGORITHM" "$f"; then
    echo "WARNING: $f has ALTER TABLE without ALGORITHM hint"
  fi
done

exit $ERRORS
```

## Best Practices

- Commit schema snapshots alongside migration files so reviewers can see the cumulative effect.
- Use `mysqldump --skip-dump-date --skip-comments` to produce clean, diff-friendly snapshots.
- Strip `AUTO_INCREMENT` values from schema snapshots since they change constantly and produce noise.
- Never modify a migration file after it has been applied to any shared environment.
- Assign version numbers at merge time to avoid numbering conflicts between branches.
- Require DBA review for any migration touching tables larger than a defined threshold (e.g., 1GB).
- Maintain separate seed data directories for reference data, dev data, and test fixtures.
- Run `pt-table-checksum` periodically to detect schema drift between primary and replicas.
- Document the purpose, author, and ticket number in every migration file header.
- Use a linter or CI check to enforce naming conventions, character set standards, and DDL safety.

## Common Mistakes

| Mistake | Impact | Fix |
|---------|--------|-----|
| Editing a migration file after it was applied to staging | Flyway/Liquibase checksum mismatch; deployment fails | Create a new migration with the correction; never modify applied migrations |
| Not stripping AUTO_INCREMENT from schema snapshots | Every commit shows a diff in the snapshot even when schema is unchanged | Use `sed 's/ AUTO_INCREMENT=[0-9]*//'` in the snapshot script |
| Using `utf8` charset in new tables instead of `utf8mb4` | Cannot store emoji or certain CJK characters; data truncation | Always specify `utf8mb4` and `utf8mb4_unicode_ci` in CREATE TABLE |
| Merging two branches with the same migration version number | One migration is silently skipped or overwritten | Use `VNEXT__` placeholder naming and renumber at merge time |
| No rollback file for a migration | Cannot reverse a failed deployment; must write rollback DDL under pressure | Require rollback files as part of the PR checklist and CI check |

## MySQL Version Notes

- **5.7**: `mysqldump` uses `utf8` as default charset. Schema snapshots may need post-processing to normalize to `utf8mb4`. No atomic DDL; failed DDL can leave partial changes.
- **8.0**: Atomic DDL for InnoDB tables ensures DDL is fully applied or fully rolled back. `information_schema` is faster (uses data dictionary cache instead of file scanning). `mysqldump` includes `--column-statistics=0` flag needed when dumping from 8.0 to load into 5.7.
- **8.4/9.x**: Data dictionary improvements make schema introspection faster. Enhanced `INFORMATION_SCHEMA` views. Deprecated `mysql_upgrade` (automatic upgrade on startup).
