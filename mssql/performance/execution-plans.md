# Execution Plans -- Reading, Analyzing, and Optimizing SQL Server Query Plans

## Overview

Execution plans are the single most important diagnostic tool for SQL Server performance tuning. When the query optimizer processes a SQL statement, it generates an execution plan that describes exactly how the database engine will retrieve, join, filter, and return data. Understanding execution plans transforms performance troubleshooting from guesswork into precise, evidence-based analysis.

Every DBA and developer working with SQL Server must be able to read execution plans fluently. Plans reveal whether indexes are being used, how tables are joined, where the optimizer expects bottlenecks, and where actual runtime behavior diverges from expectations. They surface implicit conversions, missing indexes, excessive sorts, and suboptimal join strategies that would otherwise remain invisible.

Execution plans come in two forms: estimated plans (compiled without execution) and actual plans (captured during execution with runtime statistics). The difference matters enormously -- estimated plans show what the optimizer predicts, while actual plans show what actually happened, including row count discrepancies that indicate stale statistics or cardinality estimation errors.

## Key Concepts

**Estimated Plan**: Generated by the optimizer without executing the query. Shows predicted row counts, costs, and operator choices. Useful for quick analysis but lacks runtime data.

**Actual Plan**: Captured during query execution. Includes actual row counts, memory grants, elapsed time per operator, spills to tempdb, and thread distribution for parallel plans.

**Cost-Based Optimization**: SQL Server's optimizer evaluates multiple candidate plans and selects the one with the lowest estimated cost. Cost is an internal unit (not seconds or bytes) relative to a baseline hardware profile.

**Plan Cache**: Compiled plans are stored in the plan cache for reuse. Plan reuse avoids recompilation overhead but can cause parameter sniffing issues when a cached plan is suboptimal for different parameter values.

**Operators**: Individual processing steps in an execution plan (Scan, Seek, Join, Sort, Aggregate, etc.). Data flows from right to left in graphical plans, from child operators to parent operators.

**Cardinality Estimate**: The optimizer's prediction of how many rows each operator will produce. Accurate estimates are critical for good plan quality.

## Viewing Execution Plans

### SSMS Graphical Plans

```sql
-- Estimated plan: Ctrl+L in SSMS, or:
SET SHOWPLAN_XML ON;
GO
SELECT * FROM Sales.Orders WHERE OrderDate > '2024-01-01';
GO
SET SHOWPLAN_XML OFF;
GO

-- Actual plan: Ctrl+M in SSMS to toggle "Include Actual Execution Plan"
-- Then execute the query normally

-- Text-based estimated plan (lightweight, good for scripts):
SET SHOWPLAN_ALL ON;
GO
SELECT * FROM Sales.Orders WHERE OrderDate > '2024-01-01';
GO
SET SHOWPLAN_ALL OFF;
GO

-- Text-based actual plan:
SET STATISTICS PROFILE ON;
GO
SELECT * FROM Sales.Orders WHERE OrderDate > '2024-01-01';
GO
SET STATISTICS PROFILE OFF;
GO
```

### Capturing Plans from the Plan Cache

```sql
-- Find cached plans for a specific query pattern
SELECT
    qs.execution_count,
    qs.total_elapsed_time / qs.execution_count AS avg_elapsed_us,
    qs.total_logical_reads / qs.execution_count AS avg_logical_reads,
    qs.total_worker_time / qs.execution_count AS avg_cpu_us,
    SUBSTRING(st.text, (qs.statement_start_offset / 2) + 1,
        ((CASE qs.statement_end_offset
            WHEN -1 THEN DATALENGTH(st.text)
            ELSE qs.statement_end_offset
        END - qs.statement_start_offset) / 2) + 1) AS query_text,
    qp.query_plan
FROM sys.dm_exec_query_stats AS qs
CROSS APPLY sys.dm_exec_sql_text(qs.sql_handle) AS st
CROSS APPLY sys.dm_exec_query_plan(qs.plan_handle) AS qp
WHERE st.text LIKE '%Sales.Orders%'
ORDER BY qs.total_elapsed_time DESC;

-- Find the most expensive queries in the plan cache
SELECT TOP 20
    qs.execution_count,
    qs.total_logical_reads,
    qs.total_logical_reads / qs.execution_count AS avg_reads,
    qs.total_worker_time / 1000 AS total_cpu_ms,
    qs.total_elapsed_time / 1000 AS total_elapsed_ms,
    SUBSTRING(st.text, (qs.statement_start_offset / 2) + 1,
        ((CASE qs.statement_end_offset
            WHEN -1 THEN DATALENGTH(st.text)
            ELSE qs.statement_end_offset
        END - qs.statement_start_offset) / 2) + 1) AS query_text
FROM sys.dm_exec_query_stats AS qs
CROSS APPLY sys.dm_exec_sql_text(qs.sql_handle) AS st
ORDER BY qs.total_logical_reads DESC;
```

## Reading Plan Operators

### Data Access Operators

```sql
-- Clustered Index Seek: BEST case for point lookups
-- The optimizer navigates the B-tree to find specific rows
-- Example query that produces a Clustered Index Seek:
SELECT OrderID, OrderDate, CustomerID
FROM Sales.Orders
WHERE OrderID = 42518;

-- Clustered Index Scan: Reads the ENTIRE table
-- Acceptable for small tables or queries that truly need all rows
SELECT COUNT(*) FROM Sales.Orders;

-- Nonclustered Index Seek + Key Lookup (Bookmark Lookup):
-- Seeks into a nonclustered index, then fetches remaining
-- columns from the clustered index. Expensive at high row counts.
SELECT OrderID, OrderDate, CustomerID, ShipAddress
FROM Sales.Orders
WHERE CustomerID = 1001;
-- If index is on CustomerID but ShipAddress is not included,
-- each row requires a Key Lookup back to the clustered index.

-- Table Scan: Heap table with no clustered index. Full scan required.
-- Usually a design problem -- most tables should have a clustered index.
```

### Join Operators

```sql
-- Nested Loops Join: Best for small outer input, indexed inner input
-- For each row in the outer input, seeks into the inner input
-- Cost: O(outer_rows * seek_cost)
SELECT o.OrderID, c.CustomerName
FROM Sales.Orders o
INNER JOIN Sales.Customers c ON o.CustomerID = c.CustomerID
WHERE o.OrderDate = '2024-06-15';

-- Hash Match Join: Best for large unsorted inputs with no useful index
-- Builds a hash table from the smaller input, probes with the larger
-- Cost: O(build_rows + probe_rows), requires memory grant
SELECT o.OrderID, p.ProductName
FROM Sales.OrderDetails od
INNER JOIN Sales.Products p ON od.ProductID = p.ProductID
INNER JOIN Sales.Orders o ON od.OrderID = o.OrderID;

-- Merge Join: Best for large pre-sorted inputs (or inputs with ordered index access)
-- Requires both inputs sorted on the join key
-- Cost: O(rows_input1 + rows_input2), very efficient when sorted
SELECT a.AccountID, t.TransactionAmount
FROM Accounts a
INNER JOIN Transactions t ON a.AccountID = t.AccountID;
```

### Other Key Operators

```sql
-- Sort operator: Explicit sort required when no index provides order
-- WARNING: Sorts require memory grants and can spill to tempdb
SELECT CustomerID, OrderDate, TotalAmount
FROM Sales.Orders
ORDER BY TotalAmount DESC;

-- Hash Aggregate: Groups data using a hash table
-- Used when the optimizer cannot leverage an index for grouping
SELECT CustomerID, COUNT(*), SUM(TotalAmount)
FROM Sales.Orders
GROUP BY CustomerID;

-- Stream Aggregate: Groups pre-sorted data (more efficient than Hash)
-- Used when input is already sorted on the GROUP BY columns

-- Filter: Applies a predicate that could not be pushed down
-- Often indicates a suboptimal plan

-- Compute Scalar: Evaluates expressions (usually low cost)

-- Parallelism (Gather Streams / Distribute Streams / Repartition Streams):
-- Indicates parallel plan execution across multiple threads
```

## Plan Warnings and Red Flags

### Missing Index Suggestions

```sql
-- The optimizer may suggest missing indexes in the plan XML
-- Extract missing index suggestions from the plan cache:
SELECT
    CONVERT(DECIMAL(18,2), migs.avg_total_user_cost *
        migs.avg_user_impact * (migs.user_seeks + migs.user_scans)) AS improvement_measure,
    mid.statement AS table_name,
    mid.equality_columns,
    mid.inequality_columns,
    mid.included_columns,
    migs.user_seeks,
    migs.user_scans,
    migs.avg_total_user_cost,
    migs.avg_user_impact
FROM sys.dm_db_missing_index_groups AS mig
INNER JOIN sys.dm_db_missing_index_group_stats AS migs
    ON mig.index_group_handle = migs.group_handle
INNER JOIN sys.dm_db_missing_index_details AS mid
    ON mig.index_handle = mid.index_handle
ORDER BY improvement_measure DESC;
```

### Implicit Conversions

```sql
-- Implicit conversions happen when comparing mismatched data types
-- They prevent index seeks and cause full scans
-- WRONG: varchar parameter against nvarchar column
DECLARE @SearchName VARCHAR(50) = 'Smith';
SELECT * FROM Customers WHERE LastName = @SearchName;
-- If LastName is NVARCHAR, SQL Server must convert every row

-- RIGHT: Match the data types
DECLARE @SearchName NVARCHAR(50) = N'Smith';
SELECT * FROM Customers WHERE LastName = @SearchName;
```

### Thick Arrows and Row Estimate Problems

```sql
-- Thick arrows in graphical plans indicate high row flow between operators
-- Compare "Estimated Number of Rows" vs "Actual Number of Rows"
-- A large discrepancy indicates a cardinality estimation error

-- Check for outdated statistics that cause bad estimates:
SELECT
    s.name AS stats_name,
    STATS_DATE(s.object_id, s.stats_id) AS last_updated,
    sp.rows AS table_rows,
    sp.rows_sampled,
    sp.modification_counter
FROM sys.stats AS s
CROSS APPLY sys.dm_db_stats_properties(s.object_id, s.stats_id) AS sp
WHERE s.object_id = OBJECT_ID('Sales.Orders')
ORDER BY sp.modification_counter DESC;
```

## Plan Cache Management

```sql
-- View plan cache contents and memory usage
SELECT
    objtype AS ObjectType,
    cacheobjtype AS CacheObjectType,
    COUNT(*) AS PlanCount,
    SUM(CAST(size_in_bytes AS BIGINT)) / 1024 / 1024 AS SizeMB,
    AVG(usecounts) AS AvgUseCount
FROM sys.dm_exec_cached_plans
GROUP BY objtype, cacheobjtype
ORDER BY SizeMB DESC;

-- Find single-use ad hoc plans wasting cache memory
SELECT
    COUNT(*) AS single_use_plans,
    SUM(CAST(size_in_bytes AS BIGINT)) / 1024 / 1024 AS wasted_mb
FROM sys.dm_exec_cached_plans
WHERE usecounts = 1 AND objtype = 'Adhoc';

-- Analyze plan cache by type — identify ad hoc bloat
SELECT objtype AS PlanCacheType,
    COUNT_BIG(1) AS NumOfPlans,
    SUM(CAST(size_in_bytes AS DECIMAL(18,2))) / 1024 / 1024 AS PlanSizeMB
FROM sys.dm_exec_cached_plans
GROUP BY objtype
ORDER BY NumOfPlans DESC;

-- Enable "Optimize for Ad Hoc Workloads" to reduce plan cache bloat
-- First execution: stores 384-byte plan stub instead of full 16KB+ plan
-- Second execution: stub replaced with full plan for reuse
-- Only affects newly cached plans; existing plans remain unchanged
EXEC sp_configure 'show advanced options', 1;
RECONFIGURE;
EXEC sp_configure 'optimize for ad hoc workloads', 1;
RECONFIGURE;

-- Clear the entire plan cache (use cautiously in production — causes recompilation storm)
DBCC FREEPROCCACHE;

-- Clear a specific plan from cache (preferred — surgical removal)
DBCC FREEPROCCACHE(0x06000700A421E826409CE8E5A300000001000000000000000000000000000000);

-- Find plan handle for a specific query pattern
SELECT text,
    'DBCC FREEPROCCACHE (0x' + CONVERT(VARCHAR(512), plan_handle, 2) + ')' AS freeproc_cmd
FROM sys.dm_exec_cached_plans
CROSS APPLY sys.dm_exec_sql_text(plan_handle)
WHERE text LIKE '%YourTableName%';
```

## Forced Parameterization and Plan Guides

```sql
-- Forced parameterization: Makes SQL Server parameterize all queries
-- Useful for OLTP workloads with many similar ad hoc queries
ALTER DATABASE AdventureWorks SET PARAMETERIZATION FORCED;

-- Plan guides: Force a specific plan shape for a query
EXEC sp_create_plan_guide
    @name = N'Guide_OrderLookup',
    @stmt = N'SELECT * FROM Sales.Orders WHERE CustomerID = @cid',
    @type = N'SQL',
    @module_or_batch = NULL,
    @params = N'@cid INT',
    @hints = N'OPTION (OPTIMIZE FOR (@cid = 1001))';

-- Validate plan guides
SELECT * FROM sys.plan_guides;

-- Drop a plan guide
EXEC sp_control_plan_guide N'DROP', N'Guide_OrderLookup';
```

## Plan Forcing with Query Store

```sql
-- Force a known-good plan for a regressed query
-- First, identify the query and plan IDs in Query Store
SELECT
    q.query_id,
    p.plan_id,
    p.is_forced_plan,
    rs.avg_duration,
    rs.avg_logical_io_reads,
    rs.count_executions,
    qt.query_sql_text
FROM sys.query_store_query AS q
JOIN sys.query_store_query_text AS qt ON q.query_text_id = qt.query_text_id
JOIN sys.query_store_plan AS p ON q.query_id = p.query_id
JOIN sys.query_store_runtime_stats AS rs ON p.plan_id = rs.plan_id
WHERE qt.query_sql_text LIKE '%Sales.Orders%'
ORDER BY rs.avg_duration DESC;

-- Force a specific plan
EXEC sp_query_store_force_plan @query_id = 42, @plan_id = 87;

-- Unforce a plan
EXEC sp_query_store_unforce_plan @query_id = 42, @plan_id = 87;
```

## Best Practices

- Always capture actual execution plans (not just estimated) for production troubleshooting -- actual row counts reveal cardinality estimation problems
- Look for thick arrows first -- they indicate operators processing far more rows than expected
- Check for Key Lookups on frequently executed queries -- add INCLUDE columns to eliminate them
- Watch for Sort operators with warnings -- they may be spilling to tempdb due to insufficient memory grants
- Compare estimated vs actual rows at every operator, not just the final output
- Use Query Store to track plan regressions over time rather than relying on point-in-time plan cache snapshots
- Enable "Optimize for Ad Hoc Workloads" on OLTP systems to prevent plan cache bloat from single-use queries
- Never use DBCC FREEPROCCACHE in production without understanding the recompilation storm it will cause
- Use Lightweight Query Profiling (2016 SP1+) for always-on plan diagnostics without the overhead of Extended Events

## Common Mistakes

| Mistake | Impact | Fix |
|---------|--------|-----|
| Using estimated plans for production diagnosis | Misses actual row counts, spills, and runtime warnings | Always use actual execution plans (SET STATISTICS XML ON or Ctrl+M) |
| Ignoring implicit conversion warnings | Prevents index seeks, causes full scans, inflates CPU | Match parameter data types to column types exactly |
| Adding every missing index suggestion | Index maintenance overhead, slower writes, storage bloat | Consolidate overlapping suggestions; validate with workload testing |
| Reading plan left-to-right | Misunderstanding data flow direction | Data flows right-to-left (child to parent) in graphical plans |
| Trusting cost percentages as absolute truth | Cost is a relative estimate, not actual time | Use actual elapsed time and I/O from actual plans instead |
| Clearing plan cache to "fix" performance | Massive recompilation storm, brief system-wide slowdown | Force specific plans via Query Store or plan guides |
| Ignoring parameter sniffing in cached plans | Optimal plan for one parameter value is terrible for another | Use OPTION (RECOMPILE), OPTIMIZE FOR, or Query Store |

## SQL Server Version Notes

**SQL Server 2016**: Lightweight Query Profiling (sys.dm_exec_query_profiles) available with trace flag 7412. Live Query Statistics in SSMS. Query Store introduced (2016 SP1 for Standard Edition).

**SQL Server 2017**: Adaptive Query Processing -- Batch Mode Adaptive Joins, Interleaved Execution for MSTVFs, Batch Mode Memory Grant Feedback. Automatic tuning for plan regression (with Query Store).

**SQL Server 2019**: Intelligent Query Processing suite -- Batch Mode on Rowstore, Scalar UDF Inlining, Table Variable Deferred Compilation, Memory Grant Feedback (Row Mode). Lightweight Query Profiling on by default. APPROX_COUNT_DISTINCT.

**SQL Server 2022**: Parameter Sensitive Plan optimization (multiple plans per query based on parameter values). Cardinality Estimation feedback. Degree of Parallelism feedback. Optimized Plan Forcing in Query Store. Query Store hints.
